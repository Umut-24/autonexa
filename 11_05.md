## 1. Steering servo: symmetric ±500 µs travel around 1650 µs center

### Symptom
Same `wz` produced ~2× the physical steering angle on left vs right.
Right-side parking arcs under-steered.

### Root cause
Bench-calibrated bounds were asymmetric: `us_min=1100` (550 µs swing
left) vs `us_max=1900` (250 µs swing right) around `center=1650`.

### Fix
Bridge-side bounds in `nav2_pico_bridge.py` → **1150 / 1650 / 2150**
(symmetric ±500 µs). Validation ceiling raised from 1900 to **2200** and
floor lowered to **1100** so the Diagnostics-tab param tuner can trim
*down* if a servo strains, but not run past mechanically-safe range.

Pico firmware `pico_firmware/include/config.h` is **unchanged** —
its absolute clamp `[500, 2500] µs` (in `servo.c us_to_level()`) sits far
outside the new bounds, so the firmware just forwards whatever the
bridge sends.

## 2. Single `servo_polarity` works for Nav2 *and* joystick

### Symptom
`servo_polarity=+1` → Nav2 steers correctly but the joystick is
mirrored. `servo_polarity=-1` → joystick correct, Nav2 inverted. No
single value satisfied both.

### Root cause
`ros2_mobile_bridge.publish_control()` was publishing the joystick X
axis directly to `Twist.angular.z`. Joystick convention: `+X = right
push`. ROS convention: `+angular.z = counterclockwise = left turn`.
So push-right was commanding a left turn — opposite of intuition. The
Pico-bridge's `servo_polarity` then had to be tuned for *one* of the
two consumers, and the other one was wrong.

### Fix
Invert the joystick X sign at the single emission site in the mobile
bridge:

```python
twist.angular.z = -x * max_wz * sl  # was +x
```

The calibration wizard's `calibrationPulse()` posts a `wz` argument
through the same `/api/control` endpoint, so it would have been double-
inverted. Pre-invert there too so the wizard's "turn LEFT" check still
honors the ROS sign convention:

```dart
'x': (-wz / 0.8).clamp(-1.0, 1.0),
```

- [src/parking_system/scripts/ros2_mobile_bridge.py:545-548](src/parking_system/scripts/ros2_mobile_bridge.py#L545-L548) — bridge sign flip
- [aruco_project/mobile_app/lib/services/connection_service.dart:867-877](aruco_project/mobile_app/lib/services/connection_service.dart#L867-L877) — wizard compensation

## 3. Nav2 multi-point parking maneuvers

### Symptom
Goals 15–30 cm to the side caused the car to turn full right, return
roughly to the start, then loop — never executed a proper 3-point
parallel-park.

### Root cause (multi-factor)
1. Local costmap `inflation_radius: 0.20` vs global `0.12` — the planner
   thought a goal was free but the local costmap blocked it (a pinch).
2. Robot modeled as a 10 cm circle (`robot_radius: 0.1`) while the URDF
   collision box is 30×20 cm — Nav2 underestimated the chassis length
   and overestimated the width.
3. `behavior_plugins: ["spin", "backup", "wait"]` included Spin — an
   in-place rotation is geometrically impossible on Ackermann
   (front-wheel-steer + rear-wheel-drive), so every recovery cycle
   burned one tick on a guaranteed-failure behavior.
4. RPP `max_robot_pose_search_dist: 1.0` was too short for REEDS_SHEPP
   cusps 1.5–2 m apart along the plan arc — after the first cusp RPP
   couldn't reach the next segment and re-latched onto the previous,
   producing the loop.

### Fix (in `src/parking_system/config/nav2_navigation_params.yaml`)

**Costmaps (both local + global):**
- Footprint = explicit URDF-matching rectangle (base_link is at chassis
  center per the URDF):
  ```yaml
  footprint: "[[0.15, 0.10], [0.15, -0.10], [-0.15, -0.10], [-0.15, 0.10]]"
  footprint_padding: 0.02
  ```
  `robot_radius` deleted (mutually exclusive with `footprint`).
- Local `inflation_radius: 0.20 → 0.12` to match global. Both costmaps
  now see the same drivable area.

**Planner (`GridBased`, SmacPlannerHybrid REEDS_SHEPP):**
- `minimum_turning_radius: 0.45 → 0.40 → 0.45` *(brief detour to 0.40
  reverted in follow-up — 0.40 is below the chassis's physical
  `L/tan(30°) ≈ 0.43 m`, generating un-executable plans)*.
- `reverse_penalty: 2.0 → 1.5` — makes 3-point reverse maneuvers
  competitive when geometry demands them.

**Controller (RPP):**
- `lookahead_dist: 0.40 → 0.30`, `min_lookahead_dist: 0.30 → 0.20`,
  `max_lookahead_dist: 0.60 → 0.45` — shorter carrot so RPP doesn't
  overshoot REEDS_SHEPP cusps.
- `max_robot_pose_search_dist: 1.0 → 3.0` m — RPP can find the next
  path segment across cusps that may be 1.5–2 m apart along the plan.
- `regulated_linear_scaling_min_radius: 0.50 → 0.45 → 0.50` *(detour
  reverted to match R_min restoration above)*.

**Behavior server:**
- `behavior_plugins: ["backup", "wait"]` — Spin removed. The
  corresponding `spin:` sub-block and the three `*_rotational_*`
  parameters were deleted.

**Ackermann-aware BT XML** (deferred):
- File `src/parking_system/config/bt_navigate_to_pose_ackermann.xml`
  was created (copy of Nav2's `navigate_to_pose_w_replanning_and_recovery.xml`
  with the `<Spin>` node removed from the recovery `RoundRobin`, and
  `<BackUp backup_dist="0.20"/>`).
- The launch file's `RewrittenYaml` was initially **injecting** this
  path into `bt_navigator.ros__parameters.default_nav_to_pose_bt_xml`.
- **Follow-up:** that injection was **reverted**. After commit 6958898
  Nav2 stopped publishing `/plan` and the BT swap was the only commit
  change touching the goal-accept path. The XML file remains in the
  repo, ready to opt into by re-adding one line to `param_rewrites`
  later if useful. Today bt_navigator uses the yaml's
  `default_nav_to_pose_bt_xml` (`navigate_w_replanning_only_if_path_becomes_invalid.xml`)
  which has no recovery branch at all — so the `spin` removal is
  consistent (the BT never calls it).

- [src/parking_system/config/bt_navigate_to_pose_ackermann.xml](src/parking_system/config/bt_navigate_to_pose_ackermann.xml) — staged, not active

### Ackermann reverse-steering note
The IK formula `steer = atan(L * wz / vx)` self-inverts when `vx < 0`,
so the bridge doesn't need a separate sign flip for reverse. The
existing cusp cooldown (`nav2_pico_bridge.py:445-472`) holds speed at 0
for 250 ms during a forward↔reverse transition so the servo pre-swings
before motion resumes — left untouched.

## 4. Flutter app: Desktop tab removed, Home picker, Map gesture remap,
   in-place rename

### 4a. Desktop tab deleted
The "Desktop" entry in the More-sheet mirrored a 1 Hz gnome-screenshot
of the Pi — useful when developing on the desk, dead weight in
production.

- Deleted `aruco_project/mobile_app/lib/tabs/desktop_tab.dart`.
- Removed `import 'tabs/desktop_tab.dart';` and the `_moreItem(...)` entry
  in `aruco_project/mobile_app/lib/main.dart`.
- Deleted `fetchDesktopVersion()` and `fetchDesktopShot()` from
  `connection_service.dart`.
- Bridge endpoints `/api/desktop_version` / `/api/desktop_shot` left in
  place for now — harmless if nothing polls them.

### 4b. Home tab parking-spot picker
Replaced the marker-based "Auto Park" button (which dispatched a goal
80 % of the way to the first detected ArUco marker) with a button that
opens a modal listing all saved waypoints of `kind == 'park'`.

- Button label evolved: `Auto Park` → `Park` *(commit 6958898)* →
  `Pick Park Spot` *(follow-up, after user feedback that "Park" wasn't
  obvious enough)*.
- New `_chooseParkSpot()` shows a `showModalBottomSheet` with one
  ListTile per park waypoint. Stale spots (map fingerprint mismatch)
  are disabled. Tap → `conn.navigateToNamedWaypoint(name)`.
- Old `_autoPark()` method deleted.
- Summon button kept as-is (`_summonVehicle` already handled this
  pattern for `kind == 'summon'`).

### 4c. Map tab: long-press = spot selection
**Commit 6958898 added** a long-press hit-test against
`conn.namedWaypoints` (0.15 m radius). On hit, a bottom sheet with
Navigate / Rename / Delete. On miss, the previous pose-reset dialog
still fired.

**Follow-up changed this further** — user wanted long-press to be 100 %
spot selection, no pose-reset interference:

- Hit radius bumped 0.15 m → **0.30 m** (fat-finger forgiveness).
- **On hit**: existing waypoint context menu (Navigate / Rename / Delete)
  — unchanged.
- **On miss**: new `_waypointPicker()` modal listing *all* waypoints
  (park + summon + home + custom), tap to navigate. Empty list →
  snackbar "No saved spots — add one in the Parking tab."
- **Pose-reset removed** from `_onLongPressMap` entirely.
- New entry in the ⋮ menu (`_showResetMenu`): **"Reset robot pose…"**
  opens `_resetPoseDialog`, which pre-fills the current x/y/yaw with the
  robot's pose and posts to `/api/relocalize`. Still reachable, just not
  competing with long-press.

### 4d. Parking tab: in-place rename
Each waypoint card in **Parking → Manual Spots** now has an edit icon
between the Navigate and Delete buttons. Tap → dialog with the name
prefilled. Save → POST new (under the new name) + DELETE old. The
bridge keys waypoints by `name`, so this is the documented rename path
(no new endpoint needed).

POST-first ordering means: if the save fails, the old entry stays. If
the delete fails after a successful save, a transient duplicate appears
until the 5 s poll reconciles — acceptable.
