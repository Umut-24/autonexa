% ==================== EXECUTIVE SUMMARY ====================
\chapter*{Executive Summary}
\addcontentsline{toc}{chapter}{Executive Summary}

This report presents the Critical Design Review of the \projectname{} Intelligent Parking and Recall System using the latest repository state and validation notes. The project targets autonomous indoor parking by combining live LiDAR-based mapping, ROS~2 Nav2 planning, Ackermann-compatible control, and safety-focused command arbitration. The architecture uses Raspberry Pi~5 for high-level autonomy and Raspberry Pi Pico WH for deterministic low-level actuation.

As of March~2026, core architecture maturity is estimated at approximately 80--85\%, while critical safety-command modules are functionally frozen. The command pipeline (\texttt{/cmd\_vel} $\rightarrow$ \texttt{/cmd\_vel\_smoothed} $\rightarrow$ \texttt{/cmd\_vel\_safe} $\rightarrow$ \texttt{/pico/control\_cmd}) is implemented and supported by duplicate publisher protection, timeout-to-zero behavior, and heartbeat-based supervision. Remaining work is concentrated on full physical closed-loop trials, control-source arbitration finalization, IMU-assisted fusion, and long-duration reliability testing.

% ==================== INTRODUCTION ====================
\chapter{Introduction}

\section{Project Overview}
\projectname{} is a small-scale autonomous vehicle platform designed for structured indoor parking scenarios. The system maps its local environment using LiDAR, localizes itself through SLAM and scan matching, plans a collision-aware path to a parking target, and executes safe low-speed maneuvers through a safety-filtered bridge to embedded actuators.

\section{Current Status of the Project}
Completed items include the live SLAM+Nav2 baseline, Pico bridge hardening, diagnostic scripts, and rosbag tooling. Optional EKF scaffolding without IMU has also been integrated. Partially complete items include physical e-stop characterization under motion, end-to-end autonomous field repetitions, and mode arbitration between Nav2 and mobile/manual control.

\section{Scope and Organization of the Report}
Chapter~2 explains top-down system design and subsystem interfaces. Chapter~3 documents tests and observed results. Chapter~4 summarizes cost, power, and schedule updates. Chapter~5 concludes with readiness and remaining risks. Appendices reserve space for raw logs, diagrams, and expanded plots.

% ==================== SYSTEM DESIGN ====================
\chapter{System Design}

\section{Overall System Description}
The operating principle is organized as a perception-to-actuation chain. LiDAR scans feed SLAM and localization. Nav2 generates body-frame motion commands that pass through velocity smoothing and collision monitoring before entering the Pico bridge. The bridge enforces limits and safety conditions, then transmits bounded commands to Pico firmware via micro-ROS transport.

\subsection{System-Level Block Diagram}
Figure~\ref{fig:system_block} represents the complete top-level decomposition of AutoNexa into five cooperating layers: sensing, high-level autonomy, safety-conditioned command processing, embedded real-time control, and physical actuation. The architecture intentionally splits compute responsibilities between Raspberry Pi~5 (ROS~2 and autonomy stack) and Raspberry Pi Pico WH (deterministic hardware-facing execution) so that timing-sensitive actuation does not depend on Linux scheduling jitter.

For design-review traceability, the block diagram should explicitly annotate all major boundaries and interfaces:
\begin{itemize}
    \item \textbf{Sensor ingress boundary:} LiDAR publishes raw range data on \texttt{/scan} to mapping and localization nodes.
    \item \textbf{Planning boundary:} Nav2 converts map + pose + goal into motion requests on \texttt{/cmd\_vel}.
    \item \textbf{Safety boundary:} velocity smoothing and collision monitoring reshape commands into a safe stream.
    \item \textbf{Bridge boundary:} high-level safe commands are validated and forwarded as \texttt{/pico/control\_cmd}.
    \item \textbf{Hardware boundary:} Pico translates commands into PWM/I2C outputs for steering and drive, then returns health/feedback telemetry.
\end{itemize}

Each block should include (i) owner node/module, (ii) nominal rate, and (iii) fail-safe mode. This turns the figure into both an architectural map and a verification checklist for integration and safety tests.

\subsection{Overall Data and Control Flow}
The runtime pipeline follows a closed-loop, safety-first sequence from perception to actuation:
\begin{enumerate}[label=\textbf{Step \arabic*:}, leftmargin=2.2cm]
    \item \textbf{Perception ingestion:} the LiDAR driver publishes scans on \texttt{/scan}. Frame and timestamp consistency are validated through TF before downstream consumption.
    \item \textbf{State estimation:} SLAM/localization components produce map and pose updates (map--odom--base\_link transform chain) used by navigation.
    \item \textbf{Planning and local control:} Nav2 computes a feasible trajectory and generates body-frame velocity targets on \texttt{/cmd\_vel}.
    \item \textbf{Command conditioning:} velocity smoothing enforces acceleration/jerk limits; collision monitoring attenuates or blocks unsafe forward commands near obstacles.
    \item \textbf{Safety-gated bridging:} the bridge checks command freshness, duplicate-publisher conditions, and enable state, then outputs bounded commands as \texttt{/pico/control\_cmd}.
    \item \textbf{Embedded execution:} Pico firmware converts commands into steering PWM and motor drive signals over deterministic control loops.
    \item \textbf{Feedback and supervision:} odometry, heartbeat, and enable/status topics (\texttt{/pico/heartbeat}, \texttt{/pico/enable}) flow back to ROS for monitoring, diagnostics, and operator intervention.
\end{enumerate}

Control authority is strictly forward (autonomy stack to actuator interface), while telemetry is reverse (embedded to supervisory stack). This asymmetry is deliberate: it simplifies fault containment, avoids uncontrolled control-loop recursion, and makes degraded-mode behavior easier to validate.

\section{System Architecture}

\subsection{Perception Subsystem}
The perception subsystem is LiDAR-centered and optimized for indoor parking where geometric consistency is more valuable than semantic scene understanding. It is responsible for providing stable environment representation and pose information to navigation under low-speed but high-precision maneuvering constraints.

\textbf{Primary functions:}
\begin{itemize}
    \item Publish high-integrity range scans with consistent frame IDs and timing.
    \item Maintain map quality during iterative route exploration and parking-lot layout updates.
    \item Supply localization estimates with sufficient stability for low-speed controller convergence.
\end{itemize}

\textbf{Architectural rationale:}
\begin{itemize}
    \item LiDAR-first design avoids camera fragility under illumination variation and low-texture surfaces.
    \item SLAM and localization are modular, enabling transition from live SLAM to fixed-map localization for deployment.
    \item Camera/Aruco pipelines remain optional augmentation channels, not baseline dependencies.
\end{itemize}

\subsection{Control Subsystem}
The control subsystem transforms navigation intents into physically executable commands while preserving bounded behavior under all expected faults. It is designed around low-speed conservative envelopes to prioritize safe stopping distance and predictable steering response.

\textbf{Primary functions:}
\begin{itemize}
    \item Convert body-frame velocity commands into Ackermann-compatible steering and drive references.
    \item Enforce command clamps, acceleration limits, and watchdog-based neutral fallback.
    \item Maintain deterministic command forwarding cadence toward the embedded controller.
\end{itemize}

\textbf{Safety architecture:}
\begin{itemize}
    \item Duplicate publisher checks prevent conflicting control sources from coexisting.
    \item Timeout-to-zero logic forces safe stop on stale command streams.
    \item Heartbeat and enable gating prevent silent continuation after upstream failure.
\end{itemize}

\subsection{Navigation Subsystem}
Navigation is implemented on ROS~2 Nav2 with conservative planner/controller parameters selected for repeatable indoor operation rather than aggressive traversal speed. The subsystem manages goal handling, path generation, local trajectory tracking, and recovery behavior.

\textbf{Primary functions:}
\begin{itemize}
    \item Compute collision-aware paths from current pose to designated parking targets.
    \item Generate smooth velocity profiles compatible with Ackermann kinematics and low-speed constraints.
    \item Execute recovery behaviors for local minima, short-lived scan degradation, or transient localization inconsistencies.
\end{itemize}

\textbf{Architectural rationale:}
\begin{itemize}
    \item Navfn + DWB baseline is retained due to debuggability and deterministic tuning workflow.
    \item Costmap inflation and controller gains are tuned to reduce oscillation near final parking pose.
    \item Live-map operation is favored during integration to reduce stale-map mismatch failures.
\end{itemize}

\subsection{Embedded Subsystem}
The embedded subsystem on Pico WH defines the real-time actuation boundary of the project. It receives validated high-level commands, executes hardware-timed output generation, and provides compact feedback channels to the supervisory ROS stack.

\textbf{Primary functions:}
\begin{itemize}
    \item Receive bridge commands through micro-ROS serial transport.
    \item Perform Ackermann conversion and generate steering/motor control signals (PWM/I2C/GPIO).
    \item Acquire encoder/state information and publish telemetry for diagnostics and odometry support.
\end{itemize}

\textbf{Architectural rationale:}
\begin{itemize}
    \item Real-time loops are kept on microcontroller side to isolate them from non-real-time OS behavior.
    \item Firmware-level neutral defaults on timeout provide last-resort safety independent of ROS health.
    \item Hardware abstraction in firmware decouples electrical drivers from high-level planning logic.
\end{itemize}

\subsection{Power Subsystem}
The power subsystem is partitioned to preserve compute stability during actuator transients. Logic electronics and actuation loads are supplied through separate rails, with grounding and decoupling strategy chosen to reduce noise injection into communication and sensing paths.

\textbf{Primary functions:}
\begin{itemize}
    \item Deliver regulated 5V logic power for Pi~5, Pico, and communication peripherals.
    \item Supply dedicated high-current rails for motor and steering actuators with transient headroom.
    \item Prevent brownout/reset events during rapid acceleration, braking, or steering reversals.
\end{itemize}

\textbf{Architectural rationale:}
\begin{itemize}
    \item Rail separation improves fault isolation and electrical robustness under peak load.
    \item Local decoupling and regulator margining are prioritized for worst-case current spikes.
    \item Future undervoltage telemetry hooks are preserved for long-duration reliability validation.
\end{itemize}

\section{Design Modifications}
\subsection{Modification 1: Runtime Architecture Refinement}
\textbf{Original:} static-map-dominant flow emphasis.\\
\textbf{Issue:} lower robustness during iterative bring-up and mapping updates.\\
\textbf{Modification:} live SLAM + Nav2 baseline prioritized.\\
\textbf{Justification:} improves integration observability and continuous mapping reliability.

\subsection{Modification 2: Safety Chain Hardening}
\textbf{Original:} basic bridge forwarding.\\
\textbf{Issue:} potential unsafe behavior under duplicate publishers or command loss.\\
\textbf{Modification:} lock file, graph duplicate checks, timeout-to-zero, and safe self-termination.\\
\textbf{Justification:} containment of control conflicts and safer degraded operation.

\section{Compatibility Analysis of Subsystems}
\begin{itemize}
    \item Signal interface compatibility: ROS2 topic typing and defined message contracts are enforced.
    \item Voltage compatibility: logic/actuation rails are separated to manage transients.
    \item Communication protocol compatibility: UART/USB micro-ROS path and I2C motor link are validated.
    \item Timing and latency compatibility: nominal 30 Hz command updates and timeout policy are in place.
    \item Middleware compatibility: ROS2 Jazzy, Nav2, and micro-ROS agent pipeline are aligned.
\end{itemize}

% ==================== TEST PROCEDURES ====================
\chapter{Test Procedures and Results}

\section{Test Procedures for Subsystems}

\subsection{Control Subsystem Test}
\textbf{Objective:} verify steering tracking, low-speed drive behavior, and control-path robustness.\\
\textbf{Procedure highlights:} fixed steering setpoints ($-25,-15,0,+15,+25$ deg), repeated trials, bidirectional hysteresis checks, and response timing capture.\\
\textbf{Candidate command-interface decision:} compare \texttt{TwistStamped}, wheel-speed setpoints, and normalized PWM based on integration ease, precision, and safety behavior.

\subsection{Perception Subsystem Test}
LiDAR stability validation focused on timeout fault reproduction/recovery and stable \texttt{/scan} continuity during SLAM operation.

\subsection{Communication Subsystem Test}
Tests cover launch argument contracts, bridge duplicate-publisher prevention, and topic-type integrity via diagnostic scripts.

\subsection{Power Subsystem Test}
Current stage includes estimated budgeting and rail partition validation. Extended thermal and long-duration runtime characterization remains pending.

\section{Assessment of Test Results}

\subsection{Control Subsystem Assessment}
Bridge safety behavior (timeout and duplicate detection) is validated at software integration level. Full physical closed-loop repetitions are still pending.

\subsection{Perception Subsystem Assessment}
LiDAR instability was traced to stale process ownership of \texttt{/dev/ttyUSB0}. Recovery workflow validated improved stability.

\subsection{Communication Subsystem Assessment}
Topic checks pass in strict mode. Live-flow mode fails without active command source (expected) and passes under command publication.

\subsection{Power Subsystem Assessment}
Power design is structurally adequate for current load envelope but needs extended soak tests and optional undervoltage telemetry integration.
